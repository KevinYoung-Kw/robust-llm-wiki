# File2Agent 与 Context Engineering（详细）
**中文** | [English](./05-file2agent-context-engineering.md)

## 0. 背景

1. 早期直接大批量导入知识库时，出现了“看起来处理过，但其实很多内容没读完”的问题。
2. 常见触发点是：长文件遇到读取上限、并发时读得不深、主上下文被回填过程占满。
3. 这类问题会让 entity/concept 出现遗漏，后续修复成本明显上升。

### 0.1 来自真实样本的数据锚点

1. Wiki A：`1748` 个 Markdown 文件，其中 `165` 个超过 10000 字符（`9.4%`）。
2. Wiki B：`1345` 个 Markdown 文件，其中 `126` 个超过 10000 字符（`9.4%`）。
3. 长文件是常态，不是例外：约 `9.4%` 文件超过 10000 字符。
4. `log.md` 与 `index.md` 是高风险热点：两仓库最长文件都集中在日志与索引页。
5. “文件处理过”不等于“内容读完整”：并发 + 读取上限容易造成后段遗漏。
6. 单页质量问题会沿双链扩散：新增 entity/concept 关联时，旧误差会被 `read/grep` 带入新页。
7. 工程噪声会渗入知识区：样本中出现了疑似变量未展开目录（例如 `$(...)` 片段）。

### 0.2 为什么后期会越来越烧 token（简单版）

从两个真实 Wiki 的 `entity/concept` 日增数据看，存在明显批量增长日（例如单日新增 `707`）。  
即使后面单日新增下降，累计总量仍会继续增长。

可以用一个简单公式看维护负担：

\[
Cost_t \approx \alpha \cdot \Delta N_t + \beta \cdot N_t + \gamma \cdot R_t
\]

1. \(\Delta N_t\)：当天新增的 entity/concept 数量（新建成本）。
2. \(N_t\)：到当天为止累计的 entity/concept 总量（历史体量成本）。
3. \(R_t\)：当天为建立关联与校验而触发的 `grep/read` 次数（回查成本）。

直观理解：

1. 新增页越多，当天写入成本越高。
2. 累计页越大，新增一个页面时要关联和回查的旧页越多。
3. Agent 在补链或纠错时会频繁 `grep/read` 旧内容，旧误差会被继续带入新页，回查 token 持续上升。

## 1. File2Agent（思路参考）

核心倡议：  
当一个 `raw` 文件无法在一次 `read` 中完整覆盖时，更适合让一个 subagent 对这个文件负责到底。

这样做的原因：

1. 并发处理时，Agent 可能对每个文件只读一轮，长文件后半段容易被遗漏。
2. subagent 拥有独立上下文，通常更容易把单文件读深、读全。
3. 子任务读完后把 entity/concept 候选交回主 Agent 汇总，能缓解主上下文压力。

## 2. Context Engineering（思路参考）

以下是更偏工程经验的参考方向：

1. 主 Agent 更适合做编排，subagent 更适合做单文件深读。
2. `grep` 与 `read` 可以区分职责：先定位，再精读。
3. 对长文件使用分块阅读思路，减少“前半段偏置”。
4. 在回填阶段保持结构化输出，避免原文大段进入主上下文。
5. 先保证事实和链接，再做表达层优化。

## 3. 工程化倡议（可选）

这部分不是硬规则，是为了给后续工程化提供方向：

1. 为 ingest 建立任务清单（manifest）与处理状态，方便追踪“哪些文件已完整处理”。
2. 保留阶段性中间产物（例如摘录、候选卡片、冲突标记），便于复查与回放。
3. 建立最小观测指标（如：文件读取覆盖、候选产出量、冲突量、回填后断链量）。
4. 把“主 Agent 汇总、subagent 深读”的分工作为默认协作模式。
5. 对“切片处理”增加回并约束：切片文件只作为处理中间产物，不作为最终 `src`。
6. 维护 `1 src -> 1 raw` 的映射关系：即使 raw 被切成多个片段处理，最终也要回并到切片前的原始 raw 视图。
7. 避免把 slide/分片文件直接当作独立 src 入库，否则 AI 很容易把中间产物误判为正式来源。
