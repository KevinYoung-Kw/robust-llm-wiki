# File2Agent 与 Context Engineering
**中文** | [English](./05-file2agent-context-engineering.md)

## 一句话版本

如果一个来源文件已经大到无法在一次 agent pass 里被稳妥读完，那么更安全的默认模式通常是：

`1 source file -> 1 subagent owner`

这份文档解释的，就是为什么这条规则成立、什么时候该启用，以及它和更大的上下文工程有什么关系。

## 为什么会有这条模式

大规模 wiki ingest 常见的一种问题，其实非常“安静”：

1. 某个文件被标记成“处理过”；
2. 但 agent 实际只读到了前半段；
3. 后半段的实体、概念、事实候选没有真正进入产出；
4. 等到下游页面开始出现缺口时，大家才发现遗漏。

这种失效模式往往出现在长文件遇到读取上限、并发处理读得太浅、或者主 agent 被汇总工作压满的时候。

所以这里真正要解决的，不只是 token 预算问题，而是“覆盖错觉”问题：系统看起来已经处理完成，但来源其实并没有被真正理解。

## 规则本身：一个 subagent 负责一个文件

如果单个 `raw` 文件无法在一次专注的 pass 里被稳妥覆盖，那就把它完整交给一个 subagent 负责到底。

重点不在于形式感，而在于所有权。

这个负责单文件的 subagent，至少应该承担这些职责：

1. 把文件读到足够完整，而不是只停留在前半段；
2. 提取事实、entity、concept 候选；
3. 遇到歧义和冲突时明确标记，而不是直接抹平；
4. 把结构化结果交回主 agent，由主 agent 统一合并或综合。

这样做的价值在于：长文件不容易在“只读了一半”的情况下，被系统误判为已经完成。

## 为什么文件所有权有用

### 1. 它能保护覆盖率

当一个 agent 只需要对一个文件负责时，后半段被漏掉的概率会明显下降。

### 2. 它能保护主上下文

负责编排的主 agent 可以把注意力放在路由、比较、合并决策和全库一致性上，而不是把所有原始段落都塞进一个巨大的工作记忆里。

### 3. 它让遗漏更容易暴露

一个文件只有一个清晰 owner 时，我们就更容易追问：这个来源是已经读完、只读了一部分，还是还处于未解决状态？

### 4. 它在并发下更稳定

并发本身不是问题，所有权不清才是问题。没有所有权的并发 ingest 很容易产生重复处理、局部遗漏、或者相互矛盾的半成品阅读结果。

## Context Engineering：主 Agent 应该做什么

在这个框架里，所谓上下文工程，本质上是在决定不同类型的思考应该放在哪里。

一个比较稳妥的分工通常是：

1. 主 agent 负责编排、综合和最终仓库决策；
2. subagent 负责单文件深读与局部抽取；
3. `grep` 用来定位；
4. `read` 用来确认；
5. 子任务向上返回结构化摘要，而不是把大段原文直接倒回主线程。

这点非常关键，因为原始文本是昂贵的。如果每个子任务都回传大量摘录，主编排层很快又会重新变得嘈杂、拥挤。

## 什么时候应该切块

切块是辅助策略，不是第一抽象。

以下情况适合采用分块阅读：

1. 文件太大，无法在一次细读中稳妥完成；
2. 后半段很可能包含独立事实；
3. 文件内部混杂日志、索引、叙述等多种段落，更适合分片检查。

但即使进行了切块，仓库仍然应该保持更高层的约束：

`1 src -> 1 raw`

也就是说，chunk 只是处理中间产物，不应该在知识库里长期冒充独立来源身份。

## 这份文档想预防哪些常见问题

这套建议主要是在预防下面这些反复出现的错误：

1. 把“处理过”直接等同于“读完整了”；
2. 让长文件总是偏向前半段；
3. 让切片产物被误当成规范来源；
4. 把主 agent 压得太满，导致合并判断越来越浅；
5. 让遗漏一路扩散到后续 entity、concept、synthesis 页面。

## 推荐的执行模式

对于体量稍大的 ingest 任务，比较推荐的默认协作方式是：

1. 主 agent 先枚举来源文件与风险等级；
2. 对超长或高密度文件分配专属 subagent；
3. 每个 subagent 回传结构化候选与未解问题；
4. 主 agent 负责合并、归一化、交叉检查，并写入最终 wiki 页面；
5. 页面准备进入 `stable` 之前，先跑一轮 lint。

这样一来，`File2Agent` 就不只是一个口号，而会变成真正约束覆盖质量的控制机制。

## 命名补充

对当前仓库来说，`File2Agent` 这个名字已经足够贴切，因为大多数 ingest 仍然从文件开始。

如果后面项目深入扩展到网页、API、混合来源管线，那么类似 `Source2Agent` 的名字也许会更准确。但在那之前，命名一致性比“起一个更聪明的新名字”更重要。
