# 页面属性契约
**中文** | [English](./01-attribute-proposal.md)

> 当你要回答一个很实际的问题时，请看这篇：一个生产级 wiki 页面，至少应该带哪些字段，才能让仓库长期保持可追溯、可维护？

## 1. 先给短答案

当 wiki 规模上来后，只靠正文已经不够了。

页面既要给人读，也要给 AI 读。这样一来，页面就需要一小组稳定字段，让 ingest、query 和 lint 能持续对齐：这个页面是什么、从哪来、目前是否已经可以被信任。

这就是为什么我们提出一个最小页面属性契约。

## 2. 建议的最小字段集合

一个稳定的生产页面，通常至少应带这些字段：

```yaml
id: "wk-..."
title: "..."
type: "entity|concept|topic|source|synthesis"
source_ids:
  - "src-..."
last_reviewed: "YYYY-MM-DD"
status: "draft|stable|deprecated"
topic: "..."      # 可选 / 项目自定义
category: "..."   # 可选 / 项目自定义
```

真正关键的是前六个字段。后两个更多是安全扩展示例。

## 3. 这些字段各自解决什么问题

| 字段 | 它存在的原因 |
|---|---|
| `id` | 给页面一个稳定身份，便于去重、更新和引用。 |
| `title` | 让人和 AI 都能快速定位页面。 |
| `type` | 让系统知道这是什么页面，方便分流和校验。 |
| `source_ids` | 保留 provenance，让事实可以回溯到证据。 |
| `last_reviewed` | 让内容新鲜度和过期风险变得可见。 |
| `status` | 把 draft 知识和 stable 知识区分开。 |
| `topic` | 如果项目需要，可以按主题组织页面。 |
| `category` | 给项目自定义分类预留空间。 |

## 4. 这个契约该怎么用

把这组字段当作运行基线，而不是装饰性模板。

1. 字段数量要小到团队真的愿意维护。
2. 把前六个字段当作生产页面的核心契约。
3. 只有当项目字段有清楚的 ingest 含义时，才继续往外扩。
4. 如果一个字段无法被稳定更新、稳定校验，那它大概率还不应该出现。

## 5. 为什么我们不建议给 entity/concept 写 outline template

这里最容易误解的一点是：我们并非反对模板本身；我们反对的是把模板写成“固定正文大纲”。

原因很直接：entity 或 concept 页面首先必须是一个 wiki 页面，而不是一个填空表。

如果 template 把正文 outline 写得太死，模型往往会开始生搬硬套：

1. 每一页都套同一组标题，看起来整齐，但越来越不像 wiki。
2. 双链更容易变少，因为模型会优先去填模板槽位，而不是主动组织页面之间的关系。
3. 来源容易被堆到尾部，正文内部却没有足够的 `[[wikilink]]`。
4. 页面会慢慢变成“单页自洽、但不方便跳转”的孤岛。

而 wiki 的目标不是把每一页写成同一种样子。

wiki 的目标是让知识沉淀成可维护页面，并通过双链彼此连起来。不同 entity、不同 concept，天然就应该允许有不同的展开方式。有的页面适合先给定义，有的适合先讲关系，有的适合先讲时间线。如果一开始就把正文写法锁死，最后得到的往往只是“像页面的文档”，而不是“真的能维护的 wiki”。

## 6. 为什么 template 更适合只约束笔记属性

真正适合稳定下来的，是页面属性，不是正文骨架。

原因是页面属性直接服务于系统维护：

1. AI 需要靠这些字段判断页面是什么、从哪来、现在能不能信。
2. ingest、query、lint 也需要靠这些字段保持一致口径。
3. 字段是可检查、可追溯、可批量治理的，正文 outline 往往不是。

这也是为什么我们建议：

1. template 只约束最小 frontmatter / 笔记属性；
2. 正文只保留 wiki 层面的最低要求，比如可读、可跳转、尽量有双链、避免孤岛；
3. 人读和 AI 读分层处理，而不是用一个 rigid outline 同时解决两件事。

简化理解就是：

1. AI 易读，主要靠稳定属性；
2. 人易读，主要靠自然正文；
3. Wiki 易维护，主要靠双链网络，而不是靠大纲整齐。

## 7. 为什么这件事值得先做

如果没有一个共享字段契约，同类页面很快就会写成不同形状。

然后就会出现一种非常典型的失控路径：

1. ingest 写一种结构
2. query 假设另一种结构
3. lint 只能靠反复 grep/read 去“猜”页面到底是什么意思

到最后，问题不只是 metadata 乱，而是维护成本持续上涨。

## 8. 扩展规则

当你想新增字段时，不要先开始批量写。

先回答这五件事：

1. 这个字段来自哪里
2. 它要怎么归一化
3. 没有证据时怎么办
4. 它应该在什么时候更新
5. 多来源冲突时怎么裁决
