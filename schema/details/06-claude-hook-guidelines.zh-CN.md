# Claude Hook 倡议（详细）
**中文** | [English](./06-claude-hook-guidelines.md)

## 0. 这份文档解决什么问题

1. 在 LLM-Wiki 维护中，很多错误不是“不知道怎么做”，而是“做完没及时检查”。
2. 如果把检查都留到最后，返工成本会越来越高。
3. Hook 的价值是把基础检查提前到流程中，降低遗漏和返工。

## 1. 核心定位

Hook 是全流程的轻量守门，不是某一个环节的专用功能。  
它可以服务读取、写入、回填、lint 等多个阶段。

## 2. 建议关注的触发点（思路参考）

1. 读取前触发：提示文件体量、建议分工方式（是否更适合单文件 subagent）。
2. 写入前触发：检查必要字段、来源引用、双链占位是否齐全。
3. 每次 AI 写文件后触发：做一轮快速 lint（最小检查）。
4. 批次改动后触发：做一轮范围更大的 lint（断链、命名冲突、明显冲突项）。
5. 回填 entity/concept 后触发：做 read/grep 快检，确认新旧页面关联一致。

## 3. 适合放在 Hook 里的检查

1. 结构类：字段缺失、格式明显错误、标题与类型不一致。
2. 链接类：断链、孤岛倾向、回链缺失。
3. 命名类：重复命名、同名混淆、别名冲突。
4. 基础事实类：明显时间冲突、来源缺失、同页自相矛盾。

## 4. 不建议放在 Hook 里的事情

1. 复杂长推理和大规模语义重写。
2. 高耗时的全库深度审计。
3. 强依赖人工上下文判断的结论性决策。

说明：Hook 更适合“快速提醒”和“基础守门”，不替代完整审阅。

## 5. 与 lint 的关系

1. Hook 可以做“每次写入后的快速 lint”。
2. lint 仍然需要保留“周期性全量检查”。
3. 两者关系是互补：Hook 减少坏改动进入仓库，lint 负责系统性扫描。

## 6. 与 File2Agent 的关系

1. File2Agent 解决“谁来读、怎么读”的分工问题。
2. Hook 解决“何时检查、先挡哪些错误”的流程问题。
3. 两者结合，通常能显著降低长文件 ingest 的遗漏率。
