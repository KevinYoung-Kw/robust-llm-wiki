# Turbo 模型（低参数模型）的价值（详细）
**中文** | [English](./07-turbo-model-value.md)

## 0. 背景（为什么补这一条）

在真实 Wiki 维护里，我们观察到一个稳定现象：

1. `GPT-5.3-Codex-Spark`、`GLM-5-Turbo` 这类 Turbo/低参数模型，在“创作新 wiki 页”时表现一般。
2. 但在“维护双链、检查格式、做结构一致性修正”上，表现非常好。

所以这一条不是“模型排名”，而是“任务分工倡议”。

## 1. 适合 Turbo 模型的任务

1. 双链维护：补链、断链修复、回链检查、孤页扫描后的修补。
2. 格式维护：frontmatter 字段完整性、日期格式一致性、命名规范、标题层级清理。
3. 结构 lint：`entity/concept/source/synthesis` 页的最小结构核对。
4. 批量一致化：同类页面的属性对齐、模板残留清理、重复格式噪声清理。

这类任务的共同点：  
规则清晰、目标明确、判断边界相对稳定。

## 2. 不建议只用 Turbo 模型的任务

1. 新建高质量 entity/concept 页（尤其需要跨来源整合时）。
2. 高不确定性的事实归并与冲突裁决。
3. 需要强叙事能力或深推理的 synthesis 写作。

这类任务更依赖知识综合和表达质量，不适合只靠低参数模型完成。

## 3. 推荐的分工思路（倡议）

1. 创作阶段：优先使用更稳的主模型完成事实编译与页面骨架。
2. 维护阶段：用 Turbo 模型做双链、格式、字段一致性和批量清理。
3. 回归阶段：对高风险页面再做一轮人工或高能力模型复核。

核心不是“谁更强”，而是“把模型放在更匹配的位置”。

## 4. 与 Karpathy 内核的关系

这条倡议不改变底线：

1. 仍然是 Wiki，而不是聊天记录。
2. 仍然以双链网络为核心结构。
3. 仍然服务于 `ingest -> query -> lint` 维护闭环。

Turbo 的价值在于：  
把“维护动作”做得更快、更稳、更便宜，从而让闭环更可持续。
