# Turbo 模型的价值
**中文** | [English](./07-turbo-model-value.md)

## 一句话版本

Turbo 或低成本模型，通常不是第一轮知识综合的最佳选择。

但它们往往非常适合那些规则更清楚、边界更稳定的维护工作：补链、字段修正、格式整理、批量清洗、一致性巡检，以及其他重复性的结构维护任务。

所以这份文档是一份任务路由说明，不是模型排行榜。

## 为什么要区分这两类任务

LLM-Wiki 的工作是一组风险完全不同的任务叠在一起，不是单一任务。

有些任务要求细致的事实整合、歧义处理和叙事判断；有些任务则更接近受约束的维护动作，比如修一个链接、统一一个字段、补一个标题层级、清掉一段模板残留。

如果把这两类任务混成一类，团队通常会出现两种问题：

1. 在简单维护上花了过高成本；
2. 在高风险综合上给了过低规格。

这份文档的目的，就是让模型分工和任务形状重新对齐。

## Turbo 模型通常擅长哪些工作

当任务具备这些特征时，Turbo 类模型往往是很好的选择：

1. 规则明确；
2. 重复性高；
3. 结果容易复核；
4. 即使出错，也可以通过一轮终检较低成本地纠正。

比较典型的场景包括：

1. 双链修复与回链维护；
2. frontmatter 和必填字段整理；
3. 标题层级与格式归一化；
4. 重复残留、模板噪声清理；
5. 同类页面的批量一致性检查；
6. 轻量结构 lint 辅助。

这些任务当然不是“随便做做”，但它们的判断边界通常比开放式 synthesis 清楚得多。

## 哪些任务不该默认交给 Turbo

当工作包含下面这些要求时，Turbo 模型就不应该是默认主力：

1. 从混合来源新建 entity / concept 页面；
2. 裁决互相冲突的事实；
3. 编写需要谨慎叙事的 synthesis 页面；
4. 处理身份、时间线之类的高歧义问题；
5. 生成那些将来会成为高权威参考的核心页面。

这类任务一旦出错，代价更高，因此需要更强的事实边界和推理裕量。

## 一个实用的分工方式

对这个仓库来说，一个比较健康的模式是：

1. ingest 和事实综合交给更强、或更弱幻觉的主模型；
2. 链接、格式、字段、一致性维护交给 Turbo 模型；
3. 高风险产出在稳定前，再经过人工或更强模型的复核。

这其实和整个框架里的很多设计是一致的：不要要求同一个工具在闭环的每一层都同样擅长。

## 为什么 Turbo 在战略上仍然重要

Turbo 模型的价值，不在于替代核心模型，而在于让维护闭环可以长期持续。

它们让下面这些动作在成本上更可承受：

1. 日常清理；
2. 写后守门；
3. 多轮一致性巡检；
4. 范围广但深度较浅的维护扫查。

如果这些维护动作太贵，团队就会倾向于拖延。一旦拖延，小的不一致就会逐步积累，最后削弱整座 wiki。

所以，Turbo 的战略价值并不在于“炫技”，而在于“养护”。

## 它和核心框架是什么关系

这份文档不会改变 robust LLM-Wiki 的几个基本承诺：

1. 仓库仍然是一套 wiki，而不是一堆聊天记录；
2. 双链仍然是结构的重要组成部分；
3. `ingest -> query -> lint` 仍然是主要维护闭环；
4. 更强的判断力仍然保留给高风险任务。

也正因为框架已经区分了“知识写作”和“知识结构维护”，Turbo 模型才会在这里拥有清晰而稳定的位置。
